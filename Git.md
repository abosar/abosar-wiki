# Git
## Gitとは何か
ファイルのバージョン管理ツール。業務では基本的に、チーム全体に共有されたソースコードを各々が部分的に担当・修正して完成させていく。
うっかりファイルを削除してしまっただとか、二人が同じファイルを編集して差分が反映されてないだとか、そういったトラブルを防ぐためにバージョン管理を行う。
TortoiseSVN、TFS（VisualStudio）は使用経験有り。
TFSでは編集者がロックすることで他の誰もが編集をできなくしてコンフリクト（競合）を防いでいた。
Gitでは競合した場合、責任者がどのブランチ（枝）をmainに採用するかを選ぶ。選ばれなかったブランチは残しても残さなくてもよい。

## 主なGitによるバージョン管理の流れ
1. git clone（GitHubのリポジトリを自分の作業PCにコピー）
2. コーディング
3. git add（コミットするファイルを選択）
4. git commit（編集したものを履歴に保存）
5. git pull（更新処理。競合がないか変更を確認）
6. git push

addしただけでは全体に反映されないが、commitすると全体に反映される。
英語なのでややこしいが、流れ自体はわりと単純。面倒くさいのは、コミュニケーションエラーなどで競合が発生した時。
人気のオープンソース等では競合が発生しまくるのだろうけど、業務レベルでは基本的に誰がどのソースを修正するか割り当てられているので、めったに起こらない。

## 主なGitコマンド
- git clone https://github.com/（ユーザー名）/（リポジトリ名）.git　GitHubからPCにコピー
- git status　状態確認
- git pull　更新処理
- git add .　変更ファイルを全てコミット準備
- git commit -m "コメント"　
- git push GitHubに修正を適用させる

パスの書き方がMacの場合、cd ~でホームを簡略化できたり、斜線が逆だったりするので注意。
Windows）cd C:\Users\UserName\Projects
Mac）cd ~/Projects

## .gitignoreとは
バージョン管理をしないファイル情報を書いておくファイル。
自動生成されるキャッシュファイルやデバッグログなどを指定しておくと、いちいちaddするか迷わずに済む。
